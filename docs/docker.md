# Docker Configuration

## Overview

This repository contains a **fullstack application** with a **Python backend** and a **Vue.js frontend**. The application is designed to be containerized using Docker for consistent deployment across different environments. A `Dockerfile` is provided for building the backend service image, and a separate `Dockerfile` within the `web/` directory builds the frontend assets and configures NGINX to serve them. The `docker-compose.yml` file orchestrates the startup and networking of both services, simplifying local development and deployment.

The setup leverages modern Docker practices, including:
- **Slim Base Images:** Using `python:3.11-slim` for the backend and `nginx:alpine` for the frontend to minimize image size and potential attack surface.
- **Multi-Stage Builds:** The frontend build process utilizes a multi-stage strategy to ensure that only necessary build tools (Node.js) are present in the intermediate build stage, resulting in a small final NGINX image.
- **Reproducibility:** Dependencies are managed via `requirements.txt` for Python and `package.json`/`package-lock.json` for JavaScript, ensuring consistent builds.
- **Orchestration:** `docker-compose.yml` defines the services, networks, and volume mappings, allowing for easy management of the application stack.

This documentation details how to build, run, and troubleshoot the containerized application, along with best practices for production hardening and CI/CD integration.

## Architecture Details

### Backend Service (`server.py`, `Dockerfile`)

The backend is implemented in Python and containerized using the `python:3.11-slim` base image. This choice provides a minimal footprint while ensuring compatibility with Python 3.11 features.

-   **Runtime Environment:** The `Dockerfile` likely performs the following steps:
    1.  Sets a working directory (`WORKDIR /app`).
    2.  Copies the `requirements.txt` file.
    3.  Installs Python dependencies using `pip install --no-cache-dir -r requirements.txt`. Using `--no-cache-dir` reduces image size.
    4.  Copies the rest of the application code.
    5.  Exposes the port the application listens on (e.g., `EXPOSE 8000`).
    6.  Defines the command to run the application, typically `CMD ["python", "server.py"]` for simple Flask apps or `CMD ["gunicorn", "-w", "4", "-b", "0.0.0.0:8000", "server:app"]` for production deployments using Gunicorn.
-   **Dependencies:** Managed by `requirements.txt`. The application is expected to expose an API with at least 9 endpoints, as heuristically detected.
-   **Security:** The `python:3.11-slim` image is based on Debian and is generally secure. Following best practices, the application should ideally run as a non-root user within the container.
-   **Volumes:** The container's filesystem is ephemeral. Any data that needs to persist (e.g., logs, uploaded files) must be mounted as volumes (e.g., `./logs:/app/logs`).

### Frontend Service (`web/`, `web/Dockerfile`, `web/nginx.conf`)

The frontend is a Vue.js application, built into static assets and served by NGINX.

-   **Build Process (Multi-Stage):** The `web/Dockerfile` employs a multi-stage build:
    1.  **Build Stage:** Uses a Node.js image (e.g., `node:20-alpine`) to install frontend dependencies (`npm ci` or `npm install`), copy source code, and run the build command (`npm run build`). This produces static assets, typically in a `dist/` directory.
    2.  **Production Stage:** Uses a lightweight `nginx:alpine` image. It copies the static assets from the build stage into NGINX's default web root (`/usr/share/nginx/html/`) and replaces the default NGINX configuration with `web/nginx.conf`.
-   **NGINX Configuration (`web/nginx.conf`):** This file is crucial for serving the frontend correctly:
    -   **SPA Routing:** It should include a directive like `try_files $uri $uri/ /index.html;` within the `location /` block to handle client-side routing for Single Page Applications (SPAs). This ensures that requests for routes like `/about` or `/users/1` are correctly served by `index.html`, allowing the Vue Router to take over.
    -   **Caching:** Directives for caching static assets (e.g., `expires 1y; add_header Cache-Control "public, immutable";`) are likely present to improve performance.
    -   **Compression:** Settings for `gzip` are expected to be enabled for faster asset transfer.
-   **Dependencies:** Managed by `web/package.json` and `web/package-lock.json`. The build process relies on `npm` or `yarn` commands defined in these files.
