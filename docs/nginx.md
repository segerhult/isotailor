# Nginx Configuration

## Overview

This repository implements an Nginx server as part of a fullstack application architecture, where the frontend is built using Vue 3 (via Vite) and served statically, while the backend is implemented in Python using a custom API server (`server.py`). Nginx functions in two critical roles:

1. **Static File Hosting** – serves compiled frontend assets (`index.html`, JavaScript, CSS, images) from the `web/dist/` directory (generated by Vite during builds).
2. **Reverse Proxy** – forwards API requests (e.g., `/api/*`) to the Python backend server (`server.py`) running on `localhost:8000`, decoupling frontend and backend concerns and enabling cleaner routing, CORS handling, and security policies.

The Nginx configuration is isolated within the `web/` subdirectory (`web/nginx.conf`), reflecting its role in containerized frontend deployment. This separation allows the backend to remain agnostic of HTTP server logic, while ensuring the frontend runtime environment is self-contained and portable.

## Configuration File: `web/nginx.conf`

The primary configuration file resides at [`web/nginx.conf`](web/nginx.conf). This file defines server blocks for both local development (e.g., via Docker Compose or manual invocation) and production (e.g., in orchestrated deployments such as Kubernetes or cloud VMs). Key directives include:

- `server` block listens on port `80` (HTTP) and optionally `443` (HTTPS, if TLS is enabled).
- `location /` serves static assets from the root path using `root /usr/share/nginx/html;`, assuming the compiled frontend is copied into `/usr/share/nginx/html` during container build.
- `location /api/` uses `proxy_pass` to forward requests to `http://backend:8000` (where `backend` is the Docker service name for the Python API server in `docker-compose.yml`), enabling seamless proxying without CORS errors in browsers.

> **Important**: Do *not* embed the full Nginx config in documentation. This file is the canonical source of truth for routing, and any changes should be validated directly against it.

## Containerization and Deployment

The Nginx server is containerized alongside the Vue frontend using the `web/Dockerfile`. This image is designed to be lightweight and focused: it uses `nginx:alpine` as the base image, copies compiled static assets and `nginx.conf` into the container during build, and exposes port `80`.

- **Base Image**: `nginx:alpine` (minimal attack surface, fast startup).
- **Build Integration**: Nginx is invoked automatically at container runtime via its default entrypoint; no manual start command is needed.
- **Docker Compose**: The `docker-compose.yml` file orchestrates the `nginx` service (from `web/`) and the `backend` service (Python server). Crucially, the Nginx service depends on the `backend` being healthy, and the `proxy_pass` target (`http://backend:8000`) is resolved via Docker's internal DNS.

### Example Docker Compose Service Snippet (Reference Only – See [`docker-compose.yml`](docker-compose.yml) for Full Details):
```yaml
services:
  frontend:
    build: ./web
    ports:
      - "80:80"
    depends_on:
      backend:
        condition: service_healthy

  backend:
    build: .
    ports:
      - "8000:8000"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 10s
      timeout: 5s
      retries: 3
```

## Local Development Setup

### Option 1: Development via Docker Compose
Run both frontend and backend locally with:
```bash
docker-compose up --build
```
This:
- Builds the frontend image (`web/Dockerfile`), which includes `nginx.conf` and static assets (generated at build time via `npm run build`).
- Starts Nginx in a container that proxies `/api/` to the Python backend.
- Exposes Nginx on `http://localhost`.

### Option 2: Manual Local Nginx (Advanced)
For debugging or prototyping, you may manually run Nginx *outside Docker* by:
1. Building the frontend locally:  
   ```bash
   cd web && npm install && npm run build
   ```
2. Updating `web/nginx.conf` to point to local paths:
   - Change `root /usr/share/nginx/html;` to `root /path/to/your/repo/web/dist;`
   - Change `proxy_pass http://backend:8000;` to `proxy_pass http://localhost:8000;`
3. Installing Nginx system-wide:
   - **Ubuntu/Debian**: `sudo apt-get install nginx`
   - **macOS (Homebrew)**: `brew install nginx`
4. Testing the config and starting Nginx:  
   ```bash
   nginx -t -c /path/to/web/nginx.conf
   nginx -c /path/to/web/nginx.conf
   ```

> **Note**: Avoid mixing this approach with containerized deployments in CI, as path and network assumptions differ.

## TLS Termination (Optional)

While the current `web/nginx.conf` does *not* include HTTPS configuration, the architecture is designed to support TLS termination via:
- Mounting TLS certificates (`fullchain.pem`, `privkey.pem`) into the container (e.g., via Docker secrets or volume mounts).
- Adding `listen 443 ssl;` blocks in Nginx, referencing certificate paths, and redirecting HTTP to HTTPS.

This can be enabled in production by extending the config file and adjusting `docker-compose.yml` or CI deployment scripts to inject secrets securely.

## CI/CD and Validation

- **Linter**: The GitHub Actions workflow (`.github/workflows/main.yml`) includes a step to validate `web/nginx.conf` using `nginx -t` before deployment, ensuring config syntax errors are caught early.
- **Build Pipeline**: When the frontend build step (`npm run build`) completes successfully, `web/Dockerfile` automatically bundles `dist/` and `nginx.conf`, guaranteeing consistency between frontend code and its serving configuration.

## Install Requirements

Nginx is **not installed via language-specific package managers** (e.g., pip, npm). Instead:

| Environment | Installation Method | Notes |
|-------------|---------------------|-------|
| **Production (Containerized)** | Built into `web/Dockerfile` | `FROM nginx:alpine`; no `apt-get install` needed in app code. |
| **Production (Bare Metal/VM)** | System package manager (`apt`, `yum`, `brew`) | Install `nginx` separately; config must be copied manually. |
| **Local Development (Non-Docker)** | System package manager | Same as above; must handle dependency (OpenSSL, PCRE) resolution. |
| **Docker Compose / CI** | Implicit via `nginx` base image | No explicit install step—Docker handles it in `web/Dockerfile`. |

> **Critical**: Never use `pip install nginx` or `npm install nginx`—these refer to unrelated Python/Node.js packages, not the actual Nginx HTTP server.

## Troubleshooting Tips

- **502 Bad Gateway**: Likely means Nginx cannot reach the backend (`backend:8000` unavailable or misnamed in `proxy_pass`). Verify `docker-compose.yml` service names and `healthcheck` status.
- **403 Forbidden**: Check `nginx.conf` `root` path and file permissions—ensure compiled assets exist in the expected location.
- **CORS Issues**: Ensure `location /api/` in `nginx.conf` includes `add_header Access-Control-Allow-Origin *;` *only if needed* and not duplicated by backend logic (best practice: handle CORS at the proxy layer only if backend cannot).
