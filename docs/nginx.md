# Nginx Configuration

## Overview

This repository utilizes **Nginx** as a high-performance HTTP server and reverse proxy, centralizing traffic management for a fullstack application. The application comprises a **Vue 3 frontend**, built using Vite, and a **Python-based custom API backend** served by `server.py`. Nginx acts as the gateway, receiving all incoming HTTP requests and intelligently routing them. It directly serves static assets generated by the frontend build process (`web/dist/`) and proxies dynamic API calls (e.g., requests prefixed with `/api/`) to the backend service running on port `8000`.

This architectural choice offers several key advantages:
*   **Decoupling**: It cleanly separates the frontend and backend concerns.
*   **CORS Elimination**: For browser-based clients, Nginx resolves Cross-Origin Resource Sharing issues by acting as a single origin.
*   **Centralized Control**: It provides a single point for implementing routing, security policies, and performance optimizations.
*   **Scalability**: Connection reuse, buffering, and caching capabilities enhance overall application performance and resilience.

The Vue 3 frontend, configured via `web/vite.config.js`, compiles into a set of static files (HTML, JavaScript, CSS, images) located in the `web/dist/` directory. Nginx is optimized to serve these files efficiently, leveraging techniques like memory mapping and setting aggressive client-side caching headers. When a request targets an API endpoint (e.g., `/api/users`, `/api/auth/login`), Nginx intercepts it and forwards it to the Python backend. This abstraction hides the internal network topology and allows for secure, authenticated communication without exposing internal services directly.

The deployment strategy accommodates various scenarios:
*   **Local Development**: Docker Compose provides a reproducible and isolated environment for development tasks.
*   **CI/CD Pipelines**: Workflows like `.github/workflows/main.yml` validate Nginx configuration syntax and integrity before deployment.
*   **Production Environments**: Nginx can serve as the edge server, potentially sitting behind cloud load balancers (e.g., AWS ALB). While this repository defaults to HTTP for simplicity, production deployments should strongly consider offloading TLS termination to infrastructure layers.

The file `web/nginx.conf` serves as the **single source of truth** for all Nginx behaviors within this repository. It dictates routing rules, proxy timeouts, compression settings, security headers, and Single Page Application (SPA) routing logic. All modifications to Nginx behavior must be validated using `nginx -t` prior to deployment. Documentation should focus on the *intent* and *operational impact* of the configuration, referencing `web/nginx.conf` directly rather than embedding code snippets.

## Configuration File: `web/nginx.conf`

The Nginx configuration file, located at [`web/nginx.conf`](web/nginx.conf), dictates the fundamental routing, security, and performance characteristics of the frontend service. This file is copied into the Nginx container during the image build process and becomes active upon container startup. The configuration is logically divided into distinct blocks, primarily handling static asset serving and API request proxying.

### Static Frontend Serving (`location /`)

The primary `location /` block is responsible for serving the compiled Vue application assets. It utilizes the `root` directive, specifying `/usr/share/nginx/html` as the directory containing the static files. This directory is populated during the image build by the [`web/Dockerfile`](web/Dockerfile). The `try_files` directive within this block is crucial for SPA routing:

```nginx
try_files $uri $uri/ /index.html;
```

This directive ensures that requests for existing files or directories are served directly. If no matching file or directory is found, Nginx serves `/index.html`. This allows the Vue Router to manage client-side navigation for routes like `/dashboard` or `/settings/123`. The absence of a specific `404.html` fallback here reinforces that route resolution is delegated to the frontend application.

Performance is enhanced through targeted configurations for static assets. Sub-locations (e.g., `location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2?)$`) apply aggressively long `Cache-Control` headers (`max-age=31536000`, `immutable`) and enable gzip compression. This maximizes client-side caching efficiency and minimizes bandwidth usage for frequently accessed assets.

### API Reverse Proxy (`location /api/`)

Requests targeting paths that begin with `/api/` are directed to the backend Python service using the `proxy_pass` directive:

```nginx
proxy_pass http://backend:8000;
```

In this configuration, `backend` refers to the service name defined in [`docker-compose.yml`](docker-compose.yml). Docker's internal DNS service resolves this name to the backend container's IP address. Nginx preserves critical client context by forwarding standard proxy headers:

*   `Host $host`: Transmits the original `Host` header from the client.
*   `X-Real-IP` and `X-Forwarded-For`: Relay the client's actual IP address.
*   `X-Forwarded-Proto`: Indicates whether the original client request used HTTP or HTTPS, vital for backend logic like secure redirects and CSRF protection.

Directives such as `proxy_read_timeout` are set to accommodate potentially long-running backend operations, while `proxy_connect_timeout` prevents stalled connections from impacting Nginx's performance.

### Security & Performance Enhancements

The configuration incorporates several standard optimizations:

*   **Gzip Compression**: Enabled for textual data (`gzip on; gzip_types text/plain application/json application/javascript text/css`) to reduce payload sizes.
*   **Error Handling**: Custom error pages are specified (e.g., `error_page 404 /404.html`) for a more user-friendly experience during failures.
*   **Basic Security Headers**: Headers like `X-Frame-Options` and `X-Content-Type-Options` are included for baseline security. Comprehensive OWASP-recommended headers may be added depending on specific security requirements.
*   **Rate Limiting**: The configuration includes directives for rate limiting (`limit_req_zone`), which can be enabled to mitigate brute-force attacks or denial-of-service (DoS) attempts.

**Note**: No TLS (HTTPS) configuration is present by default. For production environments, it is strongly recommended to manage TLS termination externally, typically at a load balancer or a dedicated edge proxy. If Nginx must handle TLS termination directly, refer to the [TLS Termination](#tls-termination-optional-production-enhancement) section.

> **Important**: Always consult the definitive `web/nginx.conf` for the most current configuration details. Syntax validation is essential; use `nginx -t -c web/nginx.conf` locally or the equivalent within a Docker container before committing or deploying changes.

## Containerization and Deployment

### Build Strategy for Frontend Nginx

The Nginx frontend service is containerized using [`web/Dockerfile`](web/Dockerfile). This Dockerfile is designed for efficiency and security, employing a multi-stage build pattern (though presented concisely here):

*   **Base Image**: `nginx:alpine` is selected for its minimal resource footprint (under 50MB) and robust security features. It includes comprehensive support for HTTP/2, SSL, and advanced proxying functionalities.
*   **Build-Time Frontend Compilation**: The Dockerfile executes `npm install && npm run build` during the image build process. This ensures that the `web/dist/` directory, containing the production-ready frontend assets, is generated *before* being copied into the final image. This approach guarantees deterministic builds and eliminates the need for Node.js or Vite runtime dependencies in the production environment.
*   **Configuration Embedding**: `web/nginx.conf` is copied to `/etc/nginx/conf.d/default.conf`, the standard location for custom Nginx server configurations in the Alpine image.
*   **Runtime Process**: The container uses the official Nginx image's `ENTRYPOINT ["nginx", "-g", "daemon off;"]`. This ensures Nginx runs as the primary process (PID 1), allowing it to receive and handle system signals gracefully for actions like reloads or shutdowns.

The resulting Docker image contains only the essential runtime components required for serving static files and proxying API requests, excluding any development tools, source code, or excessive logging utilities.

### Docker Compose Integration

[`docker-compose.yml`](docker-compose.yml) is used to define and manage the multi-container setup, orchestrating the Nginx frontend and the Python backend. It establishes clear dependencies and health checks:

*   **`frontend` Service**:
    *   Builds its image using the context specified in `./web` (utilizing `web/Dockerfile`).
    *   Exposes port `80` from the container to the host machine (`ports: - "80:80"`).
    *   Defines a dependency on the `backend` service, starting only *after* the `backend` has successfully passed its health check. This prevents frontend errors caused by an unavailable backend.
*   **`backend` Service**:
    *   Builds its image from the repository root (`.`) using the main [`Dockerfile`](Dockerfile).
    *   Exposes port `8000` from the container.
    *   Implements a critical health check endpoint at `/health`. Docker Compose uses `curl -f http://localhost:8000/health` to verify that this endpoint returns an `HTTP 200 OK` status code, confirming the backend application is ready to receive requests.

The `proxy_pass http://backend:8000;` directive in `web/nginx.conf` reliably resolves because Docker's internal DNS service maps the service name `backend` to the correct container IP. The default Docker network configuration provides isolation between services, enhancing security.

## Local Development Setup

### Option 1: Full Containerized Development (Recommended)

The most straightforward and reproducible method for local development relies on Docker Compose:

```bash
docker-compose up --build
```

This command automates the following sequence:

1.  **Frontend Build**: The `web/Dockerfile` is executed, performing `npm install && npm run build` to generate the `web/dist/` artifacts within the Nginx container image. The Nginx configuration is also embedded.
2.  **Backend Build & Startup**: The main Dockerfile builds the Python backend image, and the `server.py` application starts.
3.  **Service Orchestration Synchronization**: Docker Compose monitors the `backend` service. Once the `/health` endpoint on `http://localhost:8000` returns a success status code, indicating the backend is ready, the `frontend` container is started. This dependency management prevents Nginx from attempting to proxy requests to an unresponsive backend.

Upon completion, the application is accessible at `http://localhost`. The Vue application will load at the root path (`/`), and all API requests (e.g., `/api/auth/login`) will be seamlessly routed to the backend running on port `8000`. Changes made to `web/nginx.conf` require a restart of the frontend service (`docker-compose restart frontend`) to take effect.

### Option 2: Manual Local Nginx (Advanced / Debugging)

For advanced debugging or rapid testing *without* Docker Compose, Nginx can be run directly on the host system. However, this approach is **less reproducible** and generally not recommended for production or CI workflows.

#### Prerequisites
*   **Nginx Installed System-Wide**: Refer to [Install Requirements](#install-requirements) for installation instructions.
*   **Backend Running Locally**: Execute `python server.py` to start the Python backend on port `8000`.
*   **Frontend Built Locally**:
    ```bash
    cd web && npm install && npm run build
    ```

#### Temporary Configuration Adjustments

Before starting Nginx, you will need to modify `web/nginx.conf` *temporarily*. **Ensure these changes are reverted before committing**:
*   Change `root /usr/share/nginx/html;` to `root /<full_path_to_your_repo>/web/dist;` (replace `<full_path_to_your_repo>` with the actual absolute path).
*   Change `proxy_pass http://backend:8000;` to `proxy_pass http://localhost:8000;`.

#### Validation and Execution

Validate the modified configuration:
```bash
nginx -t -c /<full_path_to_your_repo>/web/nginx.conf
```

Start or reload Nginx with the adjusted configuration:
```bash
nginx -c /<full_path_to_your_repo>/web/nginx.conf
```

To apply subsequent configuration changes without stopping and restarting Nginx:
```bash
nginx -s reload
```

> **Critical Warning**: Manual local configurations are prone to drift from the canonical file managed by Docker. Always restore `web/nginx.conf` to its original state before committing code changes to ensure consistency across environments.

## TLS Termination (Optional Production Enhancement)

The default configuration assumes plain HTTP traffic. For production deployments, securing communication with **TLS (HTTPS)** is essential. The recommended practice is to **offload TLS termination** to an external service, such as a cloud load balancer (e.g., AWS ALB, Google Cloud Load Balancer) or a dedicated edge proxy (e.g., Cloudflare). This approach simplifies certificate management and reduces CPU load on the application containers.

### Scenarios for Direct TLS Termination in Nginx

Terminating TLS directly within Nginx is typically suitable for:
*   Self-hosted virtual machines or bare-metal servers that do not utilize cloud load balancing services.
*   Internal services where the complexity or cost of external TLS offloading is prohibitive.

### Configuration Requirements for Direct TLS

1.  **SSL Server Block**: A new `server` block must be added to `web/nginx.conf` (or an included configuration file) to listen on port `443` and handle SSL/TLS traffic.
    ```nginx
    server {
        listen 443 ssl http2; # Enable HTTP/2 for performance
        server_name yourdomain.com; # Replace with your domain

        ssl_certificate /etc/nginx/ssl/fullchain.pem; # Path to your certificate
        ssl_certificate_key /etc/nginx/ssl/privkey.pem; # Path to your private key

        # Include or replicate the directives for static serving and API proxying
        # from the HTTP server block.
        location / {
            root /usr/share/nginx/html;
            try_files $uri $uri/ /index.html;
        }

        location /api/ {
            proxy_pass http://backend:8000;
            # ... other proxy settings
        }

        # ... other configurations (gzip, security headers, etc.)
    }
    ```

2.  **HTTP-to-HTTPS Redirect**: A separate `server` block is needed to redirect all incoming HTTP traffic to HTTPS.
    ```nginx
    server {
        listen 80;
        server_name yourdomain.com; # Replace with your domain

        # Redirect all HTTP requests to HTTPS with a 301 Moved Permanently response
        return 301 https://$host$request_uri;
    }
    ```

3.  **Docker Compose Modifications**: To enable TLS, the Docker Compose configuration needs to be updated:
    *   **Volume Mount**: Mount a directory containing the SSL certificates into the Nginx container. Ensure the Nginx user has read permissions.
        ```yaml
        frontend:
          build: ./web
          ports:
            - "80:80"
            - "443:443" # Expose the HTTPS port
          volumes:
            - ./certs:/etc/nginx/ssl:ro # Mount certificates read-only
          # ... other frontend configurations
        ```
    *   **Port Exposure**: Both port `80` (for redirection) and `443` (for HTTPS) must be exposed.

### Certificate Management Best Practices

*   **Avoid Embedding Secrets**: Never hardcode private keys or certificates directly into Docker images. Use secure mechanisms such as:
    *   **Docker Secrets**: For Swarm or Kubernetes deployments.
    *   **CI/CD Secrets Management**: Inject secrets securely during the build or deployment process (e.g., GitHub Secrets).
    *   **Host Volume Mounting**: Mount certificates from the host system securely.
*   **Automation**: For publicly accessible services, utilize automated certificate management tools like Let's Encrypt (via `certbot`) for issuance and renewal.
*   **Internal Services**: For internal networks, use certificates issued by a trusted private Certificate Authority (CA) and distribute them via configuration management tools.

### Production Recommendation

In most production scenarios, **terminate TLS at the network edge** (e.g., using a cloud load balancer) and configure Nginx to communicate with the backend using HTTP internally. This simplifies operations, improves performance by avoiding repeated TLS handshakes in the application layer, and aligns with security best practices. If TLS is terminated upstream, `web/nginx.conf` might only need minor adjustments, such as ensuring the `X-Forwarded-Proto` header is correctly set by the upstream proxy.

## CI/CD Integration and Validation

The Continuous Integration and Continuous Deployment (CI/CD) pipeline, defined in `.github/workflows/main.yml`, includes a crucial step to validate the Nginx configuration (`web/nginx.conf`) before any deployment artifacts are built or distributed.

### Pre-Deployment Configuration Linting

A dedicated workflow job performs syntax checking on the Nginx configuration file using a reliable method:

```bash
docker run --rm -v "$(pwd)/web:/etc/nginx/conf.d" nginx:alpine nginx -t
```

This command executes the syntax check as follows:
*   It uses the official `nginx:alpine` Docker image, ensuring the `nginx` binary used for validation is consistent with the version used in the build environment.
*   It mounts only the `web/` directory into the container's Nginx configuration path (`/etc/nginx/conf.d`). This isolates the validation to the relevant configuration file and prevents potential interference from other files in the repository.
*   The `nginx -t` command performs a syntax check and reports any errors (e.g., missing semicolons, incorrect directives, invalid syntax). If errors are found, the Docker command will fail, causing the CI pipeline to halt immediately.

This pre-deployment linting step is vital for catching configuration drift that might occur during local development, merge conflicts, or automated file modifications, preventing deployment of invalid configurations.

### Build Consistency Guarantees

The CI/CD pipeline enforces a tight coupling between the Nginx configuration and the frontend build output to guarantee consistency:

1.  **Frontend Asset Generation**: The build process, typically executed within the `web/Dockerfile` during the image build phase, runs `npm run build` to generate static assets in `web/dist/`.
2.  **Atomic Copying**: Both the generated `dist/` directory and the `web/nginx.conf` file are copied into the final Nginx image. This ensures that the configuration and the assets it references are always in sync.

This approach eliminates potential mismatches such as:
*   An outdated Nginx configuration attempting to serve routes or assets that no longer exist in the latest frontend build.
*   Caching headers defined in Nginx that do not align with the naming conventions or fingerprinting of newly generated assets.

For example, if the Vite configuration (`web/vite.config.js`) changes how assets are named or output paths (e.g., from `dist/assets/file.js` to `dist/bundle.js`), the corresponding `location` block in `web/nginx.conf` (e.g., `location ~* \.(js|css)$`) must be updated accordingly. Failure to do so would result in Nginx returning `404` errors for those assets. The build-time embedding and validation process ensures such discrepancies are caught early.

## Install Requirements

Nginx is a **system-level software component** functioning as an HTTP server and reverse proxy. It is not a library installed via language-specific package managers like `pip` for Python or `npm` for JavaScript. Installing Nginx requires using the operating system's package manager or obtaining it via Docker images. Using commands like `pip install nginx` installs Python wrappers or unrelated packages, not the actual Nginx server executable, which can lead to confusion and deployment failures.

### Debugging Commands

*   **View Live Logs (Docker)**: Monitor Nginx and other services in real-time.
    ```bash
    docker-compose logs -f frontend
    # To follow all logs and grep for specific errors:
    docker-compose logs -f | grep "some error message"
    ```

*   **Inspect Container Filesystem**: Examine Nginx configuration and served files within a running container.
    ```bash
    # List files in the web root
    docker exec <frontend_container_name_or_id> ls -lR /usr/share/nginx/html

    # View the loaded Nginx configuration
    docker exec <frontend_container_name_or_id> cat /etc/nginx/conf.d/default.conf
    ```

*   **Test Backend Connectivity from Nginx Container**: Verify that the Nginx container can reach the backend service.
    ```bash
    # Check health endpoint
    ```
